# Demo Data Realism — per industry

> **INPUT CONTRACT for `threadlight-demo-data-factory`.** Per-industry rules
> for generating synthetic demo data that *feels* real to a customer SME
> without ever using real customer data.

## Why this exists

Sample data in `specs/sample-data/` is what customers see during demos. The
single fastest way to lose credibility is to ship data that an industry SME
recognizes as "obviously made up" — or worse, data that looks like it was
copied from a real customer (legal exposure).

These rules are the contract that `threadlight-demo-data-factory` uses to
seed the mock MCP server. Every industry has its own rules — these files
encode them.

## Universal rules (apply to every industry)

1. **No real PII, ever.** No real names, no real emails, no real phone
   numbers. Use Faker locales matching the geography.
2. **No real customer names — and no "Acme".** Synonym-shift well-known
   brands by **two tokens** (industry + word) so the result is plausible
   without resembling any real entity. Examples:
   - FSI: NOT "Bank of America" → "Cardinal Federal Bank" (NOT "Bankof
     America Holdings"); NOT "Boeing" → "Northwind Aviation Systems"
   - MFG: NOT "Stellantis" → "Pinnacle Automotive Group"; NOT "GE
     Aerospace" → "Cardinal Aerospace Holdings"
   - Retail: NOT "Walmart" → "Meridian General Stores"; NOT "Sephora" →
     "Solstice Beauty Co."
   - Telco: NOT "AT&T" → "Beacon Communications"; NOT "Vodafone" →
     "Northwind Telecom"

   **Banned cartoon names**: `Acme*`, `FooCorp`, `TestCo`, `Demo*`,
   `Example Inc`, `Contoso*`, `Fabrikam*`, `Northwind Traders` (verbatim
   — same string Microsoft Learn uses everywhere; instantly reads as
   tutorial). The whitelabel deny-list grep MUST include all of these.
3. **Validly-formatted, never-resolving regulatory IDs.** Use the
   officially documented "reserved-for-test" ranges from the issuing
   authority — never invent your own. Each per-industry file cites the
   specific ranges with their source. Examples:
   - US SSN: ITIN range 900–999 with middle digits 70–88/90–92/94–99 is a
     **REAL** taxpayer range — do **NOT** use it for synthetic data. Use
     `987-65-4320`–`987-65-4329` (SSA-published documentation set).
   - NANP fictional phone: `+1-NPA-555-01XX` — `555` is the central-office
     code, not the area code (per FCC / NANPA reservation).
   - SWIFT BIC: `XXX` is the *head-office* branch code, not "test". Use
     pattern `TESTBIC0XXX` or per-bank documented test BICs from the SWIFT
     reference data product.

   See per-industry files for the full citation list.
4. **Plausible distributions, not random uniform.** Skew matches reality —
   e.g. credit scores cluster around 670–720 with a long left tail; order
   values follow a power law not a uniform distribution; apparel returns
   are 25–40%, not 4%.
5. **Named "golden cases" for the demo script.** Hand-curate 3–5 records
   per entity that the seller will narrate around. These are referenced by
   name in the `specs/experience.html` storyboard. Names follow the
   two-token-shift rule above (e.g. `kyc-cardinal-aerospace`,
   `fnol-meridian-logistics-sev1`).
6. **Reset-to-pristine on demand.** Every demo data set must support an
   idempotent reset (wipe + reseed) so a botched demo run can be recovered
   in <30s. The reset script is generated by `threadlight-demo-data-factory`.

## Per-industry references

These four files cover the four threadlight pilot industries. They are NOT
the exhaustive catalog — every new industry deserves its own file in this
folder.

| File | Industry |
|------|----------|
| `fsi.md` | Financial services (banking, insurance, capital markets) |
| `retail.md` | Retail / CPG (e-commerce, brick-and-mortar, supply chain) |
| `telco.md` | Telecommunications (consumer + enterprise) |
| `mfg.md` | Manufacturing (discrete, process, supply chain) |

### Extending to a new industry

When a new threadlight process targets an industry NOT in the table above,
**add a new file** to this folder before generating demo data. Use this
checklist:

1. **Filename**: `{industry-slug}.md` (lowercase, hyphenated). Examples:
   `healthcare.md`, `public-sector.md`, `energy-utilities.md`,
   `automotive.md`, `pharma.md`, `legal.md`, `education.md`,
   `media-entertainment.md`, `travel-hospitality.md`.
2. **Required sections** (copy from `fsi.md` once it's canonized):
   - **Universal-rule deltas** — which of the 6 universal rules above need
     to be tightened or relaxed (e.g., healthcare has stricter PII rules
     than FSI; media doesn't need regulatory IDs at all)
   - **Regulatory identifier ranges** — official "documentation use" ranges
     for SSN-like, account-like, license-like identifiers in this industry
   - **Brand synonym list** — well-known brands to avoid + safe
     synonym-shifted alternatives (e.g., for healthcare: "Northwell" →
     "Northpoint Health"; for energy: "ExxonMobil" → "Apex Energy")
   - **Distribution priors** — what skewed-distribution shapes are typical
     (e.g., healthcare claim amounts follow a power law with a long tail
     for catastrophic claims)
   - **Golden case templates** — 3-5 named hand-curated cases the seller
     narrates around (e.g., healthcare: "Sarah, denied prior-auth case",
     "Marcus, complex multi-condition case", "Elena, pediatric edge case")
3. **Update spec § 11d** — once the file is added, the SPEC for any
   process targeting that industry should set `industry: <industry-slug>`
   and `data_realism_ref: references/data-realism/<industry-slug>.md`.
4. **PR back to awesome-gbb** — once the rules are battle-tested in a real
   pilot, contribute back to `threadlight-design` so future processes in
   that industry inherit the rules.

### Universal coverage limit

This folder is **NOT meant to be exhaustive across all global industries**
on day one. The realistic shape:

- 4 industries shipped today (FSI, Retail, Telco, Mfg) — drafted/placeholder
- 6-10 more added during pilots over the next 6-12 months
- Long tail (~30+ niche industries) added on-demand by whoever builds the
  first process in that industry

If you can't find a file for your industry, **DO NOT default to a neighbor
file** — the realism rules don't transfer cleanly across industry
boundaries (PII rules, regulatory IDs, distributions, vocabularies all
change). Generate a new file from the checklist above and have it
reviewed by an SME before generating any customer-facing data.

### ⚠️ HARD STOP for HIPAA-covered domains (healthcare, pharma, life-sciences)

Do **NOT** start a healthcare/pharma/life-sciences PoC by "starting from
FSI and overriding PII rules". HIPAA Privacy Rule § 164.514 (Safe Harbor
de-identification) requires removal of **18 specific identifiers** —
including dates more granular than year, ZIP-3 truncation, age caps at 89,
device identifiers — that the FSI file does not address. Adapting FSI's
PII rules creates **BAA / HIPAA exposure** if the agent or its mock data
ever come within reach of real PHI.

Required: a dedicated `healthcare.md` (or `pharma.md`) file that codifies
the 18 Safe Harbor identifiers, the Expert Determination alternative,
minimum-necessary defaults, and approved fictional-name patterns
(e.g. names from public-domain literature stripped of medical context).
Block generation if `industry ∈ {healthcare, pharma, life-sciences,
biotech, hospital, payer, provider}` and no file exists.

The same hard-stop principle applies to EU MDR, GDPR-Article-9 special
categories, and any other regime with statutory de-identification rules.

## How `threadlight-demo-data-factory` reads this

1. Spec § 11d declares industry: e.g. `industry: fsi-banking`
2. Factory loads `references/data-realism/fsi.md`
3. Factory generates per-entity Faker config inheriting universal rules + FSI overrides
4. Factory writes `scripts/seed_data.py` and `scripts/reset_data.py`
5. Hand-curated golden cases from spec § 11d are spliced in by ID
