# Fixture PoC — SPEC (excerpt for quickstart smoke tests)

This is a **minimal** Threadlight-shaped scaffold used by
`threadlight-local-test` Pattern 0 tests and walkthroughs. It is NOT a
real customer process — it's a toy that exercises the discover →
SkillsProvider → stub-tool flow end-to-end in under 15 seconds.

## Process

**Name:** support triage
**Trigger:** operator opens the demo UI; LLM is asked to "triage the
open queue"
**Outcome:** every open ticket has a severity; urgent/high tickets are
reassigned per the rules in `src/agent/skills/triage/SKILL.md`.

## §4 Data Models

| Entity | Sample data | Cardinality |
|---|---|---|
| `tickets` | `specs/sample-data/tickets.json` | 5 rows |

Ticket schema: `id`, `subject`, `description`, `status` (`open`/`closed`),
`severity` (`urgent`/`high`/`normal`/`null`), `assignee` (email or null),
`created_at` (ISO-8601).

## §5 Systems

| System | Availability | Backing |

| Support ticket DB | mock | `specs/sample-data/tickets.json` |

## §6 Tool Contracts (the convention Pattern 0 stub-tools follow)

| Tool | Reads | Writes | Generated by |
|---|---|---|---|
| `list_tickets(status=None)` | `tickets.json` | — | auto (entity = tickets) |
| `get_ticket(id)` | `tickets.json` | — | auto (entity = tickets) |
| `update_ticket(id, **fields)` | `tickets.json` | in-memory | auto (entity = tickets) |

Pattern 0's `stub_tools.py` derives the `list_/get_/update_` triple
automatically for every `<entity>.json` it finds — no per-tool wiring
needed for plain CRUD. Per-tool overrides live in
`tests/quickstart_tools.py` if the PoC needs custom predicates.

## §11d Demo Data

`tickets.json` is hand-curated so the demo always lands the same beats:
- T-1001 + T-1003 hit the "urgent" rule (outage / data loss)
- T-1005 hits the "high" rule (named tier-1 customer)
- T-1002 + T-1004 stay "normal"

Reset semantics: every quickstart launch re-reads the JSON from disk;
in-process `update_ticket` mutations are throwaway snapshots.
